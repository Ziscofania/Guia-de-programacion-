Introducción a SQL

SQL (Structured Query Language) es un lenguaje estándar para gestionar bases de datos relacionales. Permite crear, leer, actualizar y eliminar datos (CRUD), así como definir y modificar estructuras de bases de datos.

¿Qué es una base de datos relacional?

Un sistema que organiza datos en tablas relacionadas entre sí mediante claves:

Ejemplo de modelo relacional con tablas relacionadas

Componentes principales

Tablas

Estructuras que almacenan datos en filas y columnas. Cada tabla representa una entidad.

Claves

Primary Key identifica registros únicos. Foreign Key establece relaciones entre tablas.

Columnas

Atributos o campos con un tipo de dato específico (texto, número, fecha, etc.).

Registros

Filas o tuplas que representan instancias individuales de una entidad.

Historia: SQL fue desarrollado por IBM en los años 70 y se convirtió en estándar ANSI en 1986 e ISO en 1987. Aunque es un estándar, cada sistema tiene sus variantes.

Consultas SQL Básicas

Las operaciones CRUD (Create, Read, Update, Delete) son la base de SQL:

SELECT - Leer datos

SELECT * FROM clientes; -- Selecciona todos los campos de la tabla clientes

SELECT nombre, email FROM clientes; -- Selecciona solo las columnas nombre y email

SELECT DISTINCT pais FROM clientes; -- Valores únicos de la columna pais

INSERT - Crear datos

INSERT INTO clientes (nombre, email, pais)
VALUES ('Juan Pérez', 'juan@example.com', 'México'); -- Inserta un nuevo registro

INSERT INTO clientes (nombre, email, pais)
VALUES
  ('Ana Gómez', 'ana@example.com', 'España'),
  ('Carlos Ruiz', 'carlos@example.com', 'Argentina'); -- Inserta múltiples registros

UPDATE - Actualizar datos

UPDATE clientes
SET pais = 'Colombia'
WHERE id = 1; -- Actualiza el país del cliente con id 1

DELETE - Eliminar datos

DELETE FROM clientes
WHERE id = 1; -- Elimina el cliente con id 1

TRUNCATE TABLE clientes; -- Elimina todos los registros (más rápido que DELETE)

¡Cuidado! Siempre usa WHERE con UPDATE y DELETE a menos que quieras afectar TODOS los registros.

Filtros y Ordenación

El filtrado de datos se realiza con la cláusula WHERE y operadores lógicos:

Operadores de comparación

SELECT * FROM productos
WHERE precio > 100; -- Mayor que

SELECT * FROM productos
WHERE precio < 50; -- Menor que

SELECT * FROM productos
WHERE categoria = 'Electrónicos'; -- Igual a

SELECT * FROM productos
WHERE categoria != 'Electrónicos'; -- Diferente de

SELECT * FROM productos
WHERE precio BETWEEN 50 AND 100; -- Rango de valores

Operadores lógicos

SELECT * FROM productos
WHERE categoria = 'Electrónicos' AND precio > 100; -- Y lógico

SELECT * FROM productos
WHERE categoria = 'Electrónicos' OR categoria = 'Hogar'; -- O lógico

SELECT * FROM productos
WHERE NOT categoria = 'Electrónicos'; -- Negación

SELECT * FROM productos
WHERE nombre LIKE 'Televisor%'; -- Patrones: % cualquier cadena, _ un carácter

Ordenación de resultados

SELECT * FROM productos
ORDER BY precio ASC; -- Orden ascendente (default)

SELECT * FROM productos
ORDER BY precio DESC, nombre ASC; -- Ordenar por múltiples columnas

Consejo: Usa LIKE para búsquedas de texto, pero considera índices FULLTEXT para búsquedas complejas.

JOINS entre Tablas

Los JOINs permiten combinar datos de múltiples tablas relacionadas:

Visualización de los diferentes tipos de JOINs

Tipos de JOIN

-- INNER JOIN: Solo registros que coinciden en ambas tablas
SELECT p.nombre, p.precio, c.nombre AS categoria
FROM productos p
INNER JOIN categorias c ON p.categoria_id = c.id;

-- LEFT JOIN: Todos los registros de la tabla izquierda
SELECT c.nombre, COUNT(p.id) AS total_productos
FROM categorias c
LEFT JOIN productos p ON c.id = p.categoria_id
GROUP BY c.nombre;

-- RIGHT JOIN: Todos los registros de la tabla derecha
SELECT p.nombre, p.precio, c.nombre
FROM productos p
RIGHT JOIN categorias c ON p.categoria_id = c.id;

-- FULL OUTER JOIN: Todos los registros de ambas tablas
SELECT p.nombre, p.precio, c.nombre
FROM productos p
FULL OUTER JOIN categorias c ON p.categoria_id = c.id;

-- CROSS JOIN: Producto cartesiano (todas las combinaciones)
SELECT p.nombre, t.tamano
FROM productos p
CROSS JOIN tamanos t;

Alias de tablas y columnas

Los alias (AS) hacen las consultas más legibles:

SELECT
  cli.nombre AS cliente,
  ped.fecha,
  prod.nombre AS producto,
  det.cantidad,
  det.precio_unitario
FROM clientes cli
INNER JOIN pedidos ped ON cli.id = ped.cliente_id
INNER JOIN detalles_pedido det ON ped.id = det.pedido_id
INNER JOIN productos prod ON det.producto_id = prod.id
WHERE ped.fecha > '2023-01-01';

Nota: En MySQL no existe FULL OUTER JOIN, pero se puede simular con UNION de LEFT y RIGHT JOIN.

Funciones de Agregación

Permiten realizar cálculos sobre conjuntos de registros:

Funciones básicas

SELECT COUNT(*) FROM productos; -- Cuenta todos los registros

SELECT SUM(precio) FROM productos; -- Suma de valores

SELECT AVG(precio) FROM productos; -- Promedio

SELECT MIN(precio), MAX(precio) FROM productos; -- Mínimo y máximo

GROUP BY

Agrupa resultados para aplicar funciones de agregación:

SELECT categoria, COUNT(*) AS total, AVG(precio) AS precio_promedio
FROM productos
GROUP BY categoria
ORDER BY total DESC;

HAVING

Filtra grupos (similar a WHERE pero para grupos):

SELECT categoria, AVG(precio) AS precio_promedio
FROM productos
GROUP BY categoria
HAVING AVG(precio) > 100;

Ejemplo completo

SELECT
  c.nombre AS cliente,
  COUNT(p.id) AS total_pedidos,
  SUM(d.cantidad * d.precio_unitario) AS monto_total
FROM clientes c
LEFT JOIN pedidos p ON c.id = p.cliente_id
LEFT JOIN detalles_pedido d ON p.id = d.pedido_id
GROUP BY c.id, c.nombre
HAVING SUM(d.cantidad * d.precio_unitario) > 1000
ORDER BY monto_total DESC;

Consejo: WHERE filtra antes de agrupar, HAVING filtra después. Usa WHERE cuando puedas para mejor rendimiento.

Subconsultas

Consultas anidadas dentro de otras consultas, útiles para consultas complejas:

Subconsultas en WHERE

-- Productos con precio mayor al promedio
SELECT nombre, precio
FROM productos
WHERE precio > (
  SELECT AVG(precio) FROM productos
);

-- Clientes que han realizado pedidos
SELECT nombre
FROM clientes
WHERE id IN (
  SELECT DISTINCT cliente_id FROM pedidos
);

Subconsultas en FROM

-- Promedio de productos por categoría
SELECT AVG(productos_count) AS promedio
FROM (
  SELECT categoria_id, COUNT(*) AS productos_count
  FROM productos
  GROUP BY categoria_id
) AS conteo_categorias;

Subconsultas correlacionadas

-- Productos con precio mayor al promedio de su categoría
SELECT p1.nombre, p1.precio, p1.categoria_id
FROM productos p1
WHERE p1.precio > (
  SELECT AVG(p2.precio)
  FROM productos p2
  WHERE p2.categoria_id = p1.categoria_id
);

EXISTS / NOT EXISTS

-- Clientes que NO han realizado pedidos
SELECT nombre
FROM clientes c
WHERE NOT EXISTS (
  SELECT 1 FROM pedidos p
  WHERE p.cliente_id = c.id
);

Rendimiento: Las subconsultas correlacionadas pueden ser lentas en tablas grandes. Considera usar JOINs en esos casos.

DDL: Lenguaje de Definición de Datos

Comandos para crear y modificar la estructura de la base de datos:

CREATE - Crear tablas

CREATE TABLE clientes (
  id INT PRIMARY KEY AUTO_INCREMENT,
  nombre VARCHAR(100) NOT NULL,
  email VARCHAR(100) UNIQUE,
  fecha_nacimiento DATE,
  activo BOOLEAN DEFAULT TRUE,
  fecha_registro TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE pedidos (
  id INT PRIMARY KEY,
  cliente_id INT NOT NULL,
  fecha DATE NOT NULL,
  total DECIMAL(10,2),
  FOREIGN KEY (cliente_id) REFERENCES clientes(id)
);

Tipos de datos comunes

  • INT, BIGINT: Números enteros
  • DECIMAL(p,s), NUMERIC(p,s): Números precisos (p=precisión, s=escala)
  • VARCHAR(n), TEXT: Cadenas de texto
  • DATE, TIME, DATETIME, TIMESTAMP: Fechas y horas
  • BOOLEAN: Valores verdadero/falso
  • BLOB: Datos binarios

ALTER - Modificar tablas

-- Añadir columna
ALTER TABLE clientes
ADD COLUMN telefono VARCHAR(20);

-- Modificar columna
ALTER TABLE clientes
MODIFY COLUMN email VARCHAR(150);

-- Eliminar columna
ALTER TABLE clientes
DROP COLUMN telefono;

-- Añadir clave foránea
ALTER TABLE pedidos
ADD CONSTRAINT fk_cliente
FOREIGN KEY (cliente_id) REFERENCES clientes(id);

DROP - Eliminar objetos

DROP TABLE clientes; -- Elimina la tabla y sus datos

DROP DATABASE mi_db; -- Elimina la base de datos completa

Buenas prácticas: Usa migraciones para gestionar cambios en esquemas en entornos de producción. Herramientas como Flyway o Liquibase pueden ayudar.

Transacciones

Grupos de operaciones que deben ejecutarse como una sola unidad atómica:

Propiedades ACID

Atomicidad

Todas las operaciones se completan o ninguna.

Consistencia

La BD pasa de un estado válido a otro.

Aislamiento

Las transacciones no interfieren entre sí.

Durabilidad

Los cambios persisten tras completarse.

Ejemplo de transacción

START TRANSACTION;

-- 1. Actualizar saldo del cliente
UPDATE cuentas
SET saldo = saldo - 100
WHERE id = 1;

-- 2. Registrar transacción
INSERT INTO transacciones (cuenta_id, monto, tipo)
VALUES (1, 100, 'Retiro');

-- Si todo está bien, confirmar
COMMIT;

-- Si hay error, revertir
ROLLBACK;

Niveles de aislamiento

Nivel Fenómenos permitidos Rendimiento
READ UNCOMMITTED Dirty reads, Non-repeatable reads, Phantom reads Alto
READ COMMITTED Non-repeatable reads, Phantom reads Medio
REPEATABLE READ Phantom reads Medio-bajo
SERIALIZABLE Ninguno Bajo

Consejo: En aplicaciones web, usa READ COMMITTED o REPEATABLE READ como balance entre consistencia y rendimiento.

Optimización de Consultas

Técnicas para mejorar el rendimiento de tus consultas SQL:

Índices

-- Crear índice simple
CREATE INDEX idx_clientes_nombre
ON clientes(nombre);

-- Crear índice compuesto
CREATE INDEX idx_pedidos_cliente_fecha
ON pedidos(cliente_id, fecha);

-- Crear índice único
CREATE UNIQUE INDEX idx_clientes_email
ON clientes(email);

-- Eliminar índice
DROP INDEX idx_clientes_nombre ON clientes;

EXPLAIN

Muestra el plan de ejecución de una consulta:

EXPLAIN SELECT * FROM clientes
WHERE nombre = 'Juan Pérez';

Buenas prácticas

  • Índices: Crea índices en columnas usadas en WHERE, JOIN y ORDER BY
  • SELECT: Evita SELECT *, especifica solo las columnas necesarias
  • JOIN: Usa INNER JOIN en lugar de WHERE para uniones
  • Subconsultas: Conviértelas en JOINs cuando sea posible
  • LIMIT: Usa paginación (LIMIT + OFFSET) para grandes conjuntos de datos
  • Preparar consultas: Usa prepared statements para consultas repetitivas

Ejemplo de optimización

-- Antes (sin índices, con SELECT *)
SELECT * FROM pedidos
WHERE cliente_id = 5 AND fecha > '2023-01-01';

-- Después (con índices y columnas específicas)
CREATE INDEX idx_pedidos_cliente_fecha ON pedidos(cliente_id, fecha);

SELECT id, fecha, total
FROM pedidos
WHERE cliente_id = 5 AND fecha > '2023-01-01'
ORDER BY fecha DESC
LIMIT 10;

Cuidado: Los índices mejoran las lecturas pero ralentizan las inserciones/actualizaciones. No indexes en exceso.

Seguridad en SQL

Protege tu base de datos contra accesos no autorizados y ataques comunes:

SQL Injection

Ataque donde se inyecta código SQL malicioso a través de entradas de usuario:

-- Ejemplo vulnerable
SELECT * FROM usuarios
WHERE username = 'admin' AND password = '123' OR '1'='1';
-- Esto siempre devuelve todos los usuarios

Prevención

Prepared Statements

Usa parámetros en lugar de concatenar strings.

Mínimos privilegios

Asigna solo los permisos necesarios a cada usuario.

Validación de entrada

Valida y sanitiza toda entrada de usuario.

Escapar datos

Usa funciones de escape para datos dinámicos.

Ejemplo seguro (Prepared Statement)

-- En PHP usando PDO
$stmt = $pdo->prepare("SELECT * FROM usuarios WHERE username = ? AND password = ?");
$stmt->execute([$username, $password]);

-- En Node.js usando mysql2
connection.execute(
  'SELECT * FROM usuarios WHERE username = ? AND password = ?',
  [username, password],
  function(err, results) { ... }
);

Permisos y usuarios

-- Crear usuario
CREATE USER 'app_user'@'localhost' IDENTIFIED BY 'password_segura';

-- Asignar permisos
GRANT SELECT, INSERT, UPDATE ON mi_db.* TO 'app_user'@'localhost';

-- Revocar permisos
REVOKE DELETE ON mi_db.* FROM 'app_user'@'localhost';

-- Eliminar usuario
DROP USER 'app_user'@'localhost';

Importante: Nunca almacenes contraseñas en texto plano. Usa funciones de hash como bcrypt, PBKDF2 o Argon2.

SQL vs NoSQL

Comparación entre bases de datos relacionales y no relacionales:

Característica SQL NoSQL
Modelo de datos Relacional (tablas) Documentos, clave-valor, grafos, columnas
Esquema Fijo, definido de antemano Dinámico, sin esquema
Escalabilidad Vertical (más potente el servidor) Horizontal (más servidores)
Transacciones ACID completas BASE (Básicamente Disponible, Estado suave, Eventualmente consistente)
Lenguaje SQL (estándar) APIs específicas
Ejemplos MySQL, PostgreSQL, SQL Server MongoDB, Cassandra, Redis, Neo4j

¿Cuándo usar cada uno?

Usar SQL cuando...

  • Los datos son altamente estructurados
  • Necesitas integridad referencial
  • Tienes relaciones complejas entre datos
  • Requieres informes complejos

Usar NoSQL cuando...

  • Los datos son semiestructurados o no estructurados
  • Necesitas escalar horizontalmente
  • Requieres alta velocidad de escritura
  • Tu esquema cambia frecuentemente

Ejemplos de modelos

-- Modelo SQL (MySQL)
CREATE TABLE usuarios (
  id INT PRIMARY KEY,
  nombre VARCHAR(100),
  email VARCHAR(100)
);

CREATE TABLE pedidos (
  id INT PRIMARY KEY,
  usuario_id INT,
  fecha DATE,
  FOREIGN KEY (usuario_id) REFERENCES usuarios(id)
);

// Modelo NoSQL (MongoDB)
{
  "_id": ObjectId("5f8d...d4a1"),
  "nombre": "Juan Pérez",
  "email": "juan@example.com",
  "pedidos": [
    { "fecha": ISODate("2023-01-15"), "total": 125.50 },
    { "fecha": ISODate("2023-02-20"), "total": 89.99 }
  ]
}

Tendencia: Muchas aplicaciones modernas usan ambos tipos (SQL para datos transaccionales, NoSQL para contenido o analytics).
