Introducción a Git y GitHub

Git es un sistema de control de versiones distribuido creado por Linus Torvalds en 2005. Permite a los desarrolladores rastrear cambios en el código fuente durante el desarrollo de software.

GitHub, por otro lado, es una plataforma de alojamiento para proyectos que utilizan Git. Proporciona una interfaz web gráfica, herramientas de gestión de proyectos y características de colaboración como gestión de problemas y solicitudes de extracción.

Beneficios clave

  • Historial completo: Cada cambio queda registrado con autor, fecha y descripción
  • Trabajo en equipo: Múltiples desarrolladores pueden trabajar en paralelo
  • Seguridad: El código está respaldado y versionado
  • Flexibilidad: Ramas independientes para diferentes características
  • Revertir cambios: Posibilidad de volver a cualquier versión anterior

Flujo básico de trabajo con Git

Nota: Git es diferente de GitHub. Git es la herramienta de control de versiones, mientras que GitHub es un servicio que aloja repositorios Git en la nube.

Instalación de Git

Para comenzar a usar Git, primero necesitas instalarlo en tu computadora. Los pasos varían según tu sistema operativo:

Windows

  1. Descarga el instalador desde git-scm.com
  2. Ejecuta el instalador
  3. Sigue las instrucciones (opciones predeterminadas son recomendadas para principiantes)
  4. Abre Git Bash para comenzar a usar Git

macOS

Opción 1: Usa Homebrew (recomendado para desarrolladores)

brew install git

Opción 2: Descarga el instalador desde git-scm.com

Linux (Debian/Ubuntu)

sudo apt-get update
sudo apt-get install git

Verificar instalación

Para confirmar que Git se instaló correctamente, ejecuta:

git --version

Deberías ver algo como: git version 2.37.1

Configuración inicial

Después de instalar Git, es importante configurar tu identidad:

git config --global user.name "Tu Nombre"
git config --global user.email "tu@email.com"

Configuraciones útiles

Establecer el editor de texto predeterminado (VSCode en este ejemplo):

git config --global core.editor "code --wait"

Habilitar colores en la salida de Git:

git config --global color.ui auto

Ver todas tus configuraciones:

git config --list

Consejo: La configuración global se guarda en (~/.gitconfig) en Linux/macOS o C:\Users\TuUsuario\.gitconfig en Windows.

Conceptos básicos de Git

Repositorio

Un proyecto Git. Puede ser local (en tu computadora) o remoto (en un servidor como GitHub).

Commit

Un conjunto de cambios guardados en el historial con un mensaje descriptivo.

Rama (Branch)

Línea independiente de desarrollo. Por defecto, todos los repositorios tienen una rama main o master.

Working Directory

Los archivos en tu sistema de archivos que Git está monitoreando.

Staging Area (Índice)

Un área intermedia donde preparas los cambios antes de hacer commit.

HEAD

Referencia al commit actual en el que estás trabajando.

Las tres áreas principales de Git: Working Directory, Staging Area y Repository

Comandos básicos de Git

Inicializar un repositorio

Crear un nuevo repositorio Git:

git init

Clonar un repositorio existente

Descargar un repositorio remoto:

git clone https://github.com/usuario/repositorio.git

Ver el estado del repositorio

git status

Añadir archivos al staging

Añadir un archivo específico:

git add archivo.txt

Añadir todos los archivos cambiados:

git add .

Hacer commit de los cambios

git commit -m "Mensaje descriptivo del cambio"

Ver el historial de commits

git log

Versión compacta:

git log --oneline

Buenas prácticas: Los mensajes de commit deben ser descriptivos y en presente ("Agrega función X" en lugar de "Agregué función X").

Trabajo con ramas

Las ramas permiten desarrollar características de forma aislada sin afectar el código principal.

Crear una nueva rama

git branch nombre-rama

Cambiar a una rama

git checkout nombre-rama

O en versiones recientes de Git:

git switch nombre-rama

Crear y cambiar a una nueva rama

git checkout -b nombre-rama

Listar todas las ramas

git branch

Listar ramas remotas también:

git branch -a

Fusionar ramas

Primero cambia a la rama que recibirá los cambios (normalmente main):

git checkout main

Luego fusiona la otra rama:

git merge nombre-rama

Resolver conflictos

Cuando Git no puede fusionar automáticamente, marca los conflictos en los archivos. Debes editarlos manualmente, luego:

git add archivo-con-conflicto.txt
git commit

Importante: Nunca trabajes directamente en la rama main. Crea una rama para cada nueva característica o corrección.

GitHub esencial

GitHub es la plataforma más popular para alojar repositorios Git. Ofrece características adicionales como:

  • Interfaz web para repositorios
  • Pull Requests (solicitudes de extracción)
  • Issues (seguimiento de problemas)
  • GitHub Actions (CI/CD)
  • GitHub Pages (hosting estático)

Conectar un repositorio local a GitHub

  1. Crea un nuevo repositorio en GitHub (sin inicializarlo)
  2. Copia la URL del repositorio (HTTPS o SSH)
  3. En tu repositorio local, añade el remoto:
git remote add origin https://github.com/usuario/repositorio.git

Subir cambios a GitHub (primer push)

git push -u origin main

Subir cambios posteriores

git push

Descargar cambios de GitHub

git pull

Clonar un repositorio existente

git clone https://github.com/usuario/repositorio.git

Forkear un repositorio

En GitHub, hacer "fork" crea una copia personal de un repositorio en tu cuenta, permitiéndote contribuir a proyectos abiertos.

Colaboración en proyectos

Flujo de trabajo típico

  1. Hacer fork del repositorio original
  2. Clonar tu fork localmente
  3. Crear una rama para tu característica
  4. Hacer commits de tus cambios
  5. Subir la rama a tu fork
  6. Crear un Pull Request al repositorio original

Pull Request (PR)

Solicitud para fusionar tus cambios con el repositorio principal. Incluye:

  • Título descriptivo
  • Descripción detallada de los cambios
  • Referencia a issues relacionados
  • Capturas de pantalla si es necesario

Code Review

Proceso donde otros desarrolladores revisan tus cambios antes de fusionarlos. Pueden:

  • Aprobar el PR
  • Solicitar cambios
  • Dejar comentarios

Sincronizar tu fork

Para mantener tu fork actualizado con el repositorio original:

git remote add upstream https://github.com/original/repositorio.git
git fetch upstream
git merge upstream/main

Flujos de trabajo comunes

Git Flow

Modelo de ramificación estandarizado con ramas principales:

  • main/master: Código de producción
  • develop: Integración de características
  • feature/: Ramas para nuevas funcionalidades
  • release/: Preparación para producción
  • hotfix/: Correcciones urgentes

GitHub Flow

Flujo más simple adecuado para integración continua:

  1. Crea una rama desde main
  2. Haz commits en esa rama
  3. Abre un Pull Request
  4. Discute y revisa el código
  5. Despliega para pruebas
  6. Fusiona con main

Trunk-Based Development

Los desarrolladores trabajan en ramas de corta duración o directamente en main, con múltiples despliegues diarios.

Comandos avanzados de Git

Modificar el último commit

git commit --amend

Deshacer cambios

Descartar cambios en el working directory:

git checkout -- archivo.txt

Sacar archivos del staging:

git reset HEAD archivo.txt

Rebase

Reaplicar commits sobre otra rama base:

git checkout mi-rama
git rebase main

Advertencia: No hagas rebase de commits que ya han sido compartidos con otros.

Stash

Guardar cambios temporales sin hacer commit:

git stash

Recuperar cambios guardados:

git stash pop

Cherry-pick

Aplicar un commit específico de otra rama:

git cherry-pick abc1234
